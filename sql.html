Chapter 1 / 15Beginner
What is SQL?

The language that powers every data-driven decision. Before dashboards, before Python — there is SQL. This chapter builds the mental model you need before writing a single query.

~12 min Foundation Must read first
Definition
SQL (Structured Query Language, pronounced "sequel" or "S-Q-L") is a standard language for managing and querying data stored in relational databases. You use it to ask questions of your data — "how many orders came in last month?", "which users haven't logged in for 30 days?", "which product generates the most revenue?"
💡 Analogy
Imagine a massive library with millions of books, all perfectly catalogued. The database is the library. Tables are the shelves. SQL is you giving instructions to the librarian — "bring me all books written after 2010, sorted by title, only if they have more than 200 pages." The librarian fetches exactly that. You describe what you want, not how to find it.

How Databases are Organised

A relational database stores data in tables. Each table is like a spreadsheet — rows and columns — but linked to other tables and capable of holding millions of rows.

🗄️
Database
A container holding all related tables. Example: a food delivery app has one database holding users, orders, restaurants.
📋
Table
A structured grid of data with named columns and unlimited rows. A database can have dozens of tables.
🔑
Primary Key
A column that uniquely identifies every row. E.g., user_id — no two users share the same ID.
🔗
Foreign Key
A column in one table that references the primary key of another table. This links tables together.

SELECT & FROM

Every SQL query starts with SELECT. It specifies which columns you want. FROM specifies which table. Together they form the minimum valid SQL statement.

Definition
SELECT specifies which columns you want in the result. FROM specifies which table to read from. SELECT * means "all columns".
SQLbasic queries
-- Get every column and row
SELECT *
FROM users;

-- Only specific columns
SELECT name, city, platform
FROM users;

-- Rename columns using AS (alias)
SELECT
  name      AS customer_name,
  city      AS home_city,
  platform  AS device
FROM users;

-- Compute new columns inside SELECT
SELECT
  order_id,
  amount,
  amount * 0.18           AS gst,
  amount + (amount * 0.18) AS amount_with_gst
FROM orders;

-- Remove duplicate rows with DISTINCT
SELECT DISTINCT city
FROM users;
Result — SELECT DISTINCT city FROM users
city
Mumbai
Delhi
Bangalore
Pune
⚠ Avoid SELECT * in real queries
Always prefer naming specific columns. SELECT * fetches everything including columns you don't need — which is slow on large tables.
Which query renames the amount column to total_price?
SELECT total_price FROM orders
SELECT amount AS total_price FROM orders
SELECT amount = total_price FROM orders

WHERE & Filters

A query without WHERE returns every row. WHERE narrows the result to only rows you care about. It runs before grouping — it filters raw rows.

Definition
The WHERE clause filters rows from a table. Only rows where the condition is TRUE appear in the result. WHERE runs before any aggregation.

Comparison Operators

OperatorMeaningExample
=Equal tocity = 'Mumbai'
!= / <>Not equalstatus != 'cancelled'
> / <Greater / less thanamount > 500
>= / <=Greater/less or equalrating >= 4.0
SQLfilters
-- Simple equality filter
SELECT * FROM users
WHERE city = 'Mumbai';

-- AND: both conditions must be true
SELECT * FROM orders
WHERE status = 'delivered' AND amount > 500;

-- IN: cleaner than multiple OR conditions
SELECT * FROM users
WHERE city IN ('Mumbai', 'Delhi', 'Pune');

-- NOT IN: exclude values
SELECT * FROM orders
WHERE status NOT IN ('cancelled', 'refunded');

-- BETWEEN: inclusive range
SELECT * FROM orders
WHERE amount BETWEEN 300 AND 900;

-- LIKE: pattern matching
-- % = any characters,  _ = exactly one character
SELECT * FROM users WHERE name LIKE 'Ra%';      -- starts with Ra
SELECT * FROM users WHERE name LIKE '%Sharma';   -- ends with Sharma
SELECT * FROM users WHERE city  LIKE '_une';     -- 4-char city ending in une → Pune
🚫 Critical: WHERE vs HAVING
You cannot use aggregate functions like SUM(), COUNT() inside WHERE. WHERE runs on individual rows — aggregates don't exist yet. Use HAVING instead (GROUP BY chapter). This is the most common SQL error beginners make.

ORDER BY & LIMIT

Sort your results in any order. Grab only the top N rows. Without ORDER BY, SQL returns rows in no guaranteed order.

Definition
ORDER BY sorts results ascending (A→Z, 1→100) or descending (Z→A, 100→1). LIMIT caps the number of rows returned.
SQLsorting & limiting
-- Ascending (default, smallest first)
SELECT order_id, amount FROM orders
ORDER BY amount ASC;

-- Descending (largest first)
SELECT order_id, amount FROM orders
ORDER BY amount DESC;

-- Multiple columns: city A-Z, then name A-Z within each city
SELECT name, city FROM users
ORDER BY city ASC, name ASC;

-- Top 3 most expensive orders
SELECT order_id, user_id, amount
FROM orders
ORDER BY amount DESC
LIMIT 3;

-- Pagination: skip first 10, get next 10
SELECT * FROM orders
ORDER BY ordered_at DESC
LIMIT 10 OFFSET 10;
💡 Classic Pattern: Nth Highest Value
"Find the 2nd highest order amount" — one way: ORDER BY amount DESC LIMIT 1 OFFSET 1. The modern approach uses Window Functions (Chapter 13) which handles ties correctly. Learn both.

Chapter 5 / 15Intermediate
Aggregate Functions

Aggregate functions collapse many rows into a single summary value. Every dashboard metric you've ever seen is powered by one of these five functions.

Definition
An aggregate function takes a column of values and returns one number. SUM(amount) adds them all. COUNT(*) counts the rows. The five core functions: COUNT, SUM, AVG, MIN, MAX.
💡 Analogy
1,000 sales receipts in a box. COUNT — how many receipts? SUM — what's the total? AVG — what's the average bill? MIN/MAX — cheapest and most expensive? Aggregate functions answer these instantly on millions of rows.
SQLall 5 aggregates
SELECT
  COUNT(*)                  AS total_rows,       -- all rows including NULLs
  COUNT(amount)            AS non_null_amounts, -- ignores NULLs
  COUNT(DISTINCT user_id)  AS unique_customers,
  SUM(amount)              AS total_revenue,
  AVG(amount)              AS avg_order_value,
  MIN(amount)              AS cheapest_order,
  MAX(amount)              AS biggest_order
FROM orders
WHERE status = 'delivered';
⚠ COUNT(*) vs COUNT(column)
COUNT(*) counts all rows including NULLs. COUNT(column) counts only rows where that column is NOT NULL. In real datasets these give different numbers — choose intentionally.

GROUP BY & HAVING

Chapter 6 / 15Intermediate
GROUP BY & HAVING

GROUP BY is where SQL becomes a true analytical engine. Compute aggregates for every category simultaneously. HAVING filters those groups.

Definition
GROUP BY divides rows into groups based on one or more columns, then applies aggregate functions to each group separately. HAVING filters groups after aggregation.
💡 Analogy
Sort 1,000 receipts into city piles: Mumbai, Delhi, Bangalore. GROUP BY is the sorting. Then count receipts in each pile (COUNT), total them (SUM), find the biggest bill (MAX). HAVING says "show me only piles with more than 100 receipts".
SQLGROUP BY + HAVING
-- Revenue per order status
SELECT
  status,
  COUNT(*)        AS num_orders,
  SUM(amount)    AS total_revenue,
  ROUND(AVG(amount),2) AS avg_value
FROM orders
GROUP BY status;

-- HAVING: filter groups (not rows)
-- Restaurants with more than 1 delivered order
SELECT
  restaurant_id,
  COUNT(*) AS orders,
  SUM(amount) AS revenue
FROM orders
WHERE status = 'delivered'   -- WHERE filters rows first
GROUP BY restaurant_id
HAVING COUNT(*) > 1          -- HAVING filters groups after
ORDER BY revenue DESC;

SQL Execution Order — memorise this

1
FROM + JOIN
Load and combine tables.
FROM orders o JOIN users u ON o.user_id = u.user_id
2
WHERE
Filter rows. No aggregates allowed here.
WHERE status = 'delivered'
3
GROUP BY
Collapse rows into groups.
GROUP BY city
4
HAVING
Filter groups. Aggregates allowed.
HAVING COUNT(*) > 2
5
SELECT
Choose and alias columns.
SELECT city, COUNT(*) AS orders
6
ORDER BY → LIMIT
Sort and cap the final result.
ORDER BY orders DESC LIMIT 5
Which query is INVALID?
SELECT city, COUNT(*) FROM users GROUP BY city HAVING COUNT(*) > 1
SELECT city, COUNT(*) FROM users WHERE COUNT(*) > 1 GROUP BY city
SELECT city, COUNT(*) FROM users GROUP BY city ORDER BY COUNT(*) DESC

Chapter 7 / 15Intermediate
All JOIN Types

Real data lives across multiple tables. JOINs combine them so you can answer questions that span the whole database.

Definition
A JOIN combines rows from two tables based on a related column — usually a primary key matching a foreign key. Four types: INNER, LEFT, RIGHT, FULL OUTER. Each decides what happens when rows have no match in the other table.
💡 Analogy
Two lists: students and exam scores. INNER JOIN: only students who have a score. LEFT JOIN: all students, even those who didn't take the exam (score = blank). FULL OUTER JOIN: everyone from both lists.
SQLall join types
-- INNER JOIN: only matching rows
SELECT o.order_id, u.name, u.city, o.amount
FROM orders o
INNER JOIN users u  ON  o.user_id = u.user_id;

-- LEFT JOIN: all users, even without orders (NULLs for order cols)
SELECT u.name, u.city, o.order_id, o.amount
FROM users u
LEFT JOIN orders o  ON  u.user_id = o.user_id;

-- ★ Classic pattern: find users with NO orders
SELECT u.name, u.city
FROM users u
LEFT JOIN orders o  ON  u.user_id = o.user_id
WHERE o.order_id IS NULL;

-- 3 tables: orders + users + restaurants
SELECT
  o.order_id,
  u.name  AS customer,
  r.name  AS restaurant,
  o.amount
FROM orders      o
JOIN users       u  ON  o.user_id       = u.user_id
JOIN restaurants r  ON  o.restaurant_id = r.restaurant_id
WHERE o.status = 'delivered';
JOIN TypeReturnsBest for
INNER JOINOnly matched rowsCore reporting
LEFT JOINAll left + matched rightFind unmatched records
RIGHT JOINMatched left + all rightRarely used — flip tables & use LEFT
FULL OUTER JOINAll rows from bothData reconciliation

UNION & Self Join

UNION stacks rows from multiple queries. Self Join lets you compare rows within the same table.

SQLUNION & self join
-- UNION removes duplicates (slower)
SELECT city FROM users WHERE platform = 'Android'
UNION
SELECT city FROM users WHERE platform = 'iOS';

-- UNION ALL keeps all rows including duplicates (faster)
SELECT user_id, 'high_spender' AS segment FROM orders WHERE amount > 800
UNION ALL
SELECT user_id, 'low_spender'  AS segment FROM orders WHERE amount <= 400;

-- Self Join: find pairs of users in the same city
SELECT
  a.name  AS user1,
  b.name  AS user2,
  a.city
FROM users a
JOIN users b
  ON  a.city    = b.city
  AND a.user_id < b.user_id;  -- prevents duplicate pairs

Chapter 9 / 15Intermediate
Subqueries

A query inside a query. Subqueries let you use the result of one SELECT as input to another — enabling multi-step analysis in a single statement.

Definition
A subquery is a SELECT statement inside another SQL statement, enclosed in parentheses. It runs first and its result is used by the outer query.
SQLsubquery patterns
-- Subquery in WHERE: orders above average amount
SELECT order_id, amount
FROM orders
WHERE amount > (
  SELECT AVG(amount) FROM orders WHERE status = 'delivered'
);

-- Subquery in FROM (derived table)
SELECT u.name, stats.order_count, stats.total_spent
FROM users u
JOIN (
  SELECT
    user_id,
    COUNT(*)     AS order_count,
    SUM(amount) AS total_spent
  FROM orders
  WHERE status = 'delivered'
  GROUP BY user_id
) stats  ON  u.user_id = stats.user_id;

-- EXISTS: users with at least one delivered order
SELECT u.name, u.city
FROM users u
WHERE EXISTS (
  SELECT 1 FROM orders o
  WHERE o.user_id = u.user_id
    AND o.status  = 'delivered'
);
✦ Subquery vs CTE
Subqueries in FROM get hard to read when nested deeply. CTEs (Chapter 12) are the modern preferred way — they name each step clearly. Whenever you write a subquery in FROM, consider whether a CTE would be cleaner.

CASE WHEN

SQL's if-else. Create conditional columns, bucket values into categories, and pivot data — all within a single query.

Definition
CASE WHEN evaluates conditions top to bottom and returns the value of the first THEN whose condition is true. If nothing matches, returns the ELSE value (or NULL).
SQLCASE WHEN patterns
-- Categorise orders by amount
SELECT
  order_id, amount,
  CASE
    WHEN amount <  300 THEN 'Budget'
    WHEN amount <  700 THEN 'Mid-range'
    WHEN amount < 1200 THEN 'Premium'
    ELSE                    'Luxury'
  END AS tier
FROM orders;

-- CASE inside aggregation — conditional pivot
-- Count delivered vs cancelled per restaurant in one query
SELECT
  restaurant_id,
  COUNT(*)  AS total,
  COUNT(CASE WHEN status = 'delivered'  THEN 1 END) AS delivered,
  COUNT(CASE WHEN status = 'cancelled'  THEN 1 END) AS cancelled,
  ROUND(
    COUNT(CASE WHEN status = 'delivered' THEN 1 END) * 100.0
    / COUNT(*), 1
  ) AS success_pct
FROM orders
GROUP BY restaurant_id;

Handling NULLs

NULL is not zero, not empty string — it means "unknown". Mishandling NULLs is one of the most common sources of wrong analysis results.

Definition
NULL means the absence of a value. Any arithmetic with NULL returns NULL. NULL = NULL is NULL (not TRUE). This is why you must use IS NULL, not = NULL.
SQLnull handling
-- WRONG: this never returns rows
SELECT * FROM orders WHERE amount = NULL;

-- CORRECT
SELECT * FROM orders WHERE amount IS NULL;
SELECT * FROM orders WHERE amount IS NOT NULL;

-- COALESCE: return first non-NULL value
SELECT
  order_id,
  COALESCE(amount, 0)  AS amount_or_zero
FROM orders;

-- Multiple fallbacks
COALESCE(mobile, landline, email, 'no contact')

-- NULLIF: returns NULL if two values are equal
-- Prevents division by zero safely
SUM(amount) / NULLIF(COUNT(*), 0)  -- safe division

CTEs — WITH Clause

Common Table Expressions turn tangled nested subqueries into clean, step-by-step logic. The upgrade every SQL writer needs.

Definition
A CTE is a named temporary result set defined with the WITH keyword at the top of a query. You reference it by name as if it were a real table. CTEs exist only for that query's duration.
💡 Analogy
CTEs are like mise en place in cooking — prep each ingredient separately, name it, then assemble the dish. Instead of one messy step, each component is clean, testable, and reusable.
SQLCTE patterns
-- Basic CTE: cleaner than nested subquery
WITH delivered_stats AS (
  SELECT
    user_id,
    SUM(amount) AS total_spent,
    COUNT(*)    AS order_count
  FROM orders
  WHERE status = 'delivered'
  GROUP BY user_id
)
SELECT u.name, u.city, ds.total_spent, ds.order_count
FROM users u
JOIN delivered_stats ds  ON  u.user_id = ds.user_id
ORDER BY ds.total_spent DESC;

-- Multiple CTEs chained (each can reference the previous)
WITH
good_orders AS (
  SELECT * FROM orders WHERE status = 'delivered'
),
user_totals AS (
  SELECT user_id, SUM(amount) AS ltv
  FROM good_orders          -- ← references CTE above
  GROUP BY user_id
),
segments AS (
  SELECT user_id, ltv,
    CASE
      WHEN ltv >= 1000 THEN 'Gold'
      WHEN ltv >=  500 THEN 'Silver'
      ELSE                   'Bronze'
    END AS tier
  FROM user_totals
)
SELECT u.name, s.tier, s.ltv
FROM users u
JOIN segments s  ON  u.user_id = s.user_id
ORDER BY s.ltv DESC;

Chapter 13 / 15Advanced
Window Functions

The most powerful feature in analytical SQL. Compute rankings, running totals, and period-over-period comparisons while keeping every row intact.

Definition
A window function computes a value over a set of rows related to the current row — its "window" — using the OVER() clause. Unlike GROUP BY, window functions add a computed column while keeping every row.
💡 GROUP BY vs Window
10 exam scores. GROUP BY + AVG: returns 1 number — the class average. AVG() OVER(): returns 10 rows, each with the original score plus the class average as a new column. Individual and aggregate together.
SQLwindow functions
-- ROW_NUMBER, RANK, DENSE_RANK
-- Difference: how they handle tied values
-- Scores 100,90,90,80 → ROW_NUMBER: 1,2,3,4 | RANK: 1,2,2,4 | DENSE_RANK: 1,2,2,3
SELECT
  order_id, amount,
  ROW_NUMBER()  OVER (ORDER BY amount DESC)  AS row_num,
  RANK()        OVER (ORDER BY amount DESC)  AS rank_val,
  DENSE_RANK()  OVER (ORDER BY amount DESC)  AS dense_rank
FROM orders;

-- PARTITION BY: restart ranking per group
RANK() OVER (
  PARTITION BY restaurant_id   -- separate rank per restaurant
  ORDER BY     amount DESC
)

-- ★ TOP N PER GROUP — the most important pattern
-- Each user's single best order
WITH ranked AS (
  SELECT
    user_id, order_id, amount,
    ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY     amount DESC
    ) AS rn
  FROM orders WHERE status = 'delivered'
)
SELECT user_id, order_id, amount
FROM ranked
WHERE rn = 1;

-- LAG: compare to previous row
LAG(amount, 1, 0) OVER (
  PARTITION BY user_id
  ORDER BY     ordered_at
) AS prev_order_amount

-- Running total per user
SUM(amount) OVER (
  PARTITION BY user_id
  ORDER BY     ordered_at
) AS running_total
✦ Quick Reference
ROW_NUMBER() unique row numbers   RANK() rank with gaps   DENSE_RANK() rank no gaps
LAG(col,n) n rows back   LEAD(col,n) n rows ahead
SUM() OVER() running/group total   AVG() OVER() moving average   NTILE(n) bucket into n groups

String & Date Functions

Clean messy text and work with dates directly inside SQL. Essential for real-world data that is never perfectly formatted.

SQLstring & date functions
-- String functions
UPPER(name)                        -- "RAJESH KUMAR"
LOWER(name)                        -- "rajesh kumar"
TRIM(name)                         -- remove leading/trailing spaces
LENGTH(name)                       -- character count
SUBSTRING(name, 1, 6)             -- "Rajesh"
REPLACE(city, 'Bangalore', 'Bengaluru')
CONCAT(name, ' (', city, ')')     -- "Rajesh Kumar (Mumbai)"
SPLIT_PART(name, ' ', 1)          -- "Rajesh" (PostgreSQL)

-- Date functions (PostgreSQL)
DATE(ordered_at)                    -- "2024-04-01"
EXTRACT(YEAR  FROM ordered_at)    -- 2024
EXTRACT(MONTH FROM ordered_at)    -- 4
EXTRACT(DOW   FROM ordered_at)    -- 0=Sun...6=Sat
TO_CHAR(ordered_at, 'YYYY-MM')    -- "2024-04" (month label)

-- Date filters
WHERE ordered_at >= CURRENT_DATE - INTERVAL '30 days'
WHERE EXTRACT(MONTH FROM ordered_at) = 4   -- April only
⚠ SQL Dialect Differences
This guide uses PostgreSQL syntax. MySQL uses YEAR(col), MONTH(col). SQL Server uses DATEPART(month, col). BigQuery: EXTRACT(MONTH FROM col). Concepts are identical — only function names differ.

Chapter 15 / 15Practice
Practice Problems

Six problems spanning the full range. Attempt each one before revealing the solution. The explanation is as important as the code.

P1 — Users Who Never Ordered
easy
Find all users who have never placed any order. Return name, city, platform. Sort by name.
Hint: Which JOIN type keeps all users even without matching orders?
SQL
SELECT u.name, u.city, u.platform
FROM users u
LEFT JOIN orders o  ON  u.user_id = o.user_id
WHERE o.order_id IS NULL
ORDER BY u.name;
LEFT JOIN keeps every user. For users with no orders, o.order_id is NULL. WHERE filters to only those NULL rows = users with zero orders. The "LEFT JOIN + IS NULL" pattern is one of the most commonly used in analytics.
P2 — City Revenue Report
medium
For each city, calculate total revenue from delivered orders, number of unique customers, and average order value. Show only cities with more than 1 delivered order. Sort by revenue descending.
SQL
SELECT
  u.city,
  SUM(o.amount)              AS total_revenue,
  COUNT(DISTINCT o.user_id)   AS unique_customers,
  ROUND(AVG(o.amount),2)     AS avg_order_value
FROM orders o
JOIN users u  ON  o.user_id = u.user_id
WHERE o.status = 'delivered'
GROUP BY u.city
HAVING COUNT(*) > 1
ORDER BY total_revenue DESC;
WHERE filters rows first. GROUP BY collapses by city. HAVING removes cities with ≤1 order. COUNT(DISTINCT user_id) ensures one user ordering 3 times counts as 1 customer.
P3 — Second Highest Order Amount
medium
Find the second highest order amount. Handle ties correctly — if two orders share the top amount, the 2nd highest is the next distinct value.
SQL
-- DENSE_RANK handles ties correctly (preferred)
WITH ranked AS (
  SELECT amount,
    DENSE_RANK() OVER (ORDER BY amount DESC) AS dr
  FROM orders
)
SELECT DISTINCT amount AS second_highest
FROM ranked WHERE dr = 2;
DENSE_RANK gives tied rows the same rank and doesn't skip the next rank. So even if two orders share rank 1, the next distinct value correctly gets rank 2.
P4 — Platform Delivery Success Rate
medium
For each platform, calculate total orders, delivered count, and delivery success rate % (1 decimal place).
SQL
SELECT
  u.platform,
  COUNT(*)  AS total_orders,
  COUNT(CASE WHEN o.status = 'delivered' THEN 1 END) AS delivered,
  ROUND(
    COUNT(CASE WHEN o.status = 'delivered' THEN 1 END) * 100.0
    / COUNT(*), 1
  ) AS success_pct
FROM orders o
JOIN users u  ON  o.user_id = u.user_id
GROUP BY u.platform
ORDER BY success_pct DESC;
CASE WHEN inside COUNT returns 1 only for delivered rows, NULL for others. COUNT ignores NULLs — so it counts only delivered orders. Multiply by 100.0 (not 100) to force decimal division.
P5 — Month-over-Month Revenue Change
hard
Calculate total delivered revenue per month and the month-over-month % change from the previous month.
SQL
WITH monthly AS (
  SELECT
    TO_CHAR(ordered_at, 'YYYY-MM') AS month,
    SUM(amount)                  AS revenue
  FROM orders
  WHERE status = 'delivered'
  GROUP BY 1
)
SELECT
  month, revenue,
  LAG(revenue) OVER (ORDER BY month) AS prev_revenue,
  ROUND(
    (revenue - LAG(revenue) OVER (ORDER BY month)) * 100.0
    / NULLIF(LAG(revenue) OVER (ORDER BY month), 0),
  1) AS mom_change_pct
FROM monthly
ORDER BY month;
CTE aggregates revenue per month. LAG() fetches the previous month's value. NULLIF prevents division by zero for the first month. This CTE + LAG pattern is the standard for period-over-period metrics.
P6 — Retained Users Across Two Months
hard
Find users who placed at least one delivered order in both April 2024 AND May 2024. Return name, city, and order counts for each month.
SQL
WITH
apr AS (
  SELECT user_id, COUNT(*) AS apr_orders
  FROM orders
  WHERE status = 'delivered'
    AND EXTRACT(YEAR  FROM ordered_at) = 2024
    AND EXTRACT(MONTH FROM ordered_at) = 4
  GROUP BY user_id
),
may AS (
  SELECT user_id, COUNT(*) AS may_orders
  FROM orders
  WHERE status = 'delivered'
    AND EXTRACT(YEAR  FROM ordered_at) = 2024
    AND EXTRACT(MONTH FROM ordered_at) = 5
  GROUP BY user_id
)
SELECT u.name, u.city, a.apr_orders, m.may_orders
FROM apr a
JOIN may   m  ON  a.user_id = m.user_id
JOIN users u  ON  a.user_id = u.user_id;
Build one CTE per month. INNER JOIN finds the intersection — users in both. This cohort intersection pattern is the backbone of retention analysis.
